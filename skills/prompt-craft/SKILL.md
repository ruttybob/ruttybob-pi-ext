# Prompt Craft — создание goal-промптов для pi

Навык помогает писать промпты для pi-агента — файлы формата `goal_*.md`, которые задают
структурированный рабочий процесс с автоматическим управлением целями через встроенные
инструменты.

## Формат промптов проекта

Все промпты хранятся в `prompts/` и состоят из двух частей:

1. **YAML frontmatter** — метаданные для pi
2. **Тело** — инструкции для агента

```markdown
---
description: Краткое описание — что делает промпт
argument-hint: "<аргументы-для-пользователя>"
---

Текст инструкций для агента...
```

Поля frontmatter:
- `description` (обязательный) — что промпт делает, отображается в списке промптов
- `argument-hint` (опциональный) — подсказка аргументов, показывается в UI
- `preset` (опциональный) — название пресета настроек агента

Подстановка аргументов:
- `$@` — весь текст аргумента, переданный пользователем
- `$1`, `$2`... — позиционные аргументы

## Инструменты goal

pi предоставляет три встроенных инструмента для управления целями. Промпты `goal_*.md`
автоматически инструктируют агента использовать их для самостоятельного контроля хода
работы.

### create_goal

Создаёт новую активную цель. Только одна цель может быть активна одновременно.

```
create_goal({
  objective: "Цель — что именно нужно сделать",
  token_budget: 150000  // опционально, положительное число
})
```

**Когда вызывать:** В начале работы, если задача требует нескольких шагов и проверки
результата. Агент должен сформулировать конкретную цель с чёткими критериями завершения
перед тем как начать работу.

**Хорошая цель** содержит:
- Конкретный scope — какие файлы, модули, директории затрагиваются
- Критерии успеха — как выглядит "готово" (тесты проходят, линтер чист, файлы созданы)
- Ограничения — бюджет, стоп-условия, чего избегать
- Верификацию — как подтвердить правильность результата

**Плохая цель:** "Почини код"
**Хорошая цель:** "Рефакторинг модуля src/auth/: заменить все прямые вызовы fetch на
httpClient из src/lib/http.ts. Обновить юнит-тесты. Запустить тест-сьют и убедиться,
что 0 падений. Стоп при первом упавшем тесте."

### update_goal

Отмечает цель как завершённую. Единственный допустимый статус — `"complete"`.

```
update_goal({ status: "complete" })
```

**Когда вызывать:** Только после completion audit — проверки, что все требования цели
действительно выполнены. Агент не должен отмечать цель завершённой, если хотя бы одно
требование не проверено.

**Completion audit** — перед вызовом update_goal агент обязан:
1. Переформулировать цель как конкретные артефакты
2. Сопоставить каждое требование с реальными доказательствами (файлы, тест-вывод)
3. Проверить покрытие — прохождение тестов само по себе не доказывает завершение
4. Найти и перечислить отсутствующие или неполные элементы
5. Только если всё проверено — вызвать update_goal

### get_goal

Возвращает текущее состояние цели — статус, токены, бюджет, количество ходов.

```
get_goal()
```

**Когда вызывать:**
- В начале нового хода — понять, где мы находимся
- Перед update_goal — убедиться, что цель ещё активна
- В середине длинной задачи — проверить оставшийся бюджет

Ответ содержит:
- `goal` — объект с objective, status, tokensUsed, tokenBudget, turnCount
- `remainingTokens` — сколько токенов осталось в бюджете (undefined если бюджет не задан)

## Структура goal-промпта

Хороший `goal_*.md` промпт следует такому шаблону:

```markdown
---
description: Описание — что промпт делает и когда использовать
argument-hint: "<аргументы>"
---

<Описание задачи для агента>

<goal>
$@
</goal>

## Workflow

1. Прочитай задачу пользователя
2. Вызови create_goal с конкретной целью и бюджетом
3. Выполни работу пошагово:
   - [шаг 1]
   - [шаг 2]
   - ...
4. Перед завершением — проведи completion audit
5. Вызови update_goal({ status: "complete" })

## Правила

- [правило 1]
- [правило 2]

## Вывод

Что сохранять и куда:
\`\`\`
.plans/<slug>/YYYY-MM-DD_HHMMSS-<slug>.md
\`\`\`

После сохранения открой файл:
\`\`\`bash
open -a mods <plan>
\`\`\`
```

Блок `<goal>$@</goal>` передаёт аргумент пользователя в секцию goal — это стандартный
паттерн промптов проекта.

## Бюджет

Токен-бюджет — основной рычаг контроля стоимости. Ориентиры для цели:

| Масштаб задачи | Файлы | Бюджет (токенов) |
|---|---|---|
| Малая | ~20 | 50 000 – 150 000 |
| Средняя | ~50 | 150 000 – 500 000 |
| Крупная | 100+ | 500 000 – 2 000 000+ |

Бюджет лучше ставить консервативно — всегда можно продолжить доработку.

## Примеры

### goal_plan.md — планирование с целью

```markdown
---
description: Summary plan with goal tracking
argument-hint: "<task-description>"
---

Create a plan for the task. Do NOT implement anything.

<goal>
$@
</goal>

## Workflow

1. Read the user's request
2. Call create_goal with a concrete objective derived from the task:
   - Scope: which files and modules
   - Success: the plan is complete, well-structured, and saved
   - Budget: 100000 tokens
3. Explore the codebase with read and bash for context
4. Write the plan
5. Call get_goal to verify status, then update_goal({ status: "complete" })

## Plan must include

- Goal — one sentence
- Context — current state, constraints
- Approach — architecture decision
- Tasks — ordered, concrete steps
- Files to change — exact paths
- Risks — what could go wrong

## Output

Save to:
.plans/<slug>/YYYY-MM-DD_HHMMSS-<slug>.md

Then open with:
open -a mods <plan>
```

### goal_refactor.md — рефакторинг с целью

```markdown
---
description: Refactor with behavior preservation and goal tracking
argument-hint: "<path-or-module>"
---

Refactor the specified code preserving all behavior.

<goal>
$@
</goal>

## Workflow

1. Identify the scope of refactoring
2. Call create_goal:
   - Objective: specific refactoring with measurable outcome
   - Budget: 200000
3. Before each change, verify current behavior with tests
4. Make one refactoring pass at a time
5. Run tests after each pass
6. Completion audit: all tests pass, no behavior change
7. update_goal({ status: "complete" })

## Rules

- Preserve behavior unless explicitly asked for functional change
- One refactoring pass per step
- Run tests after each change
- If a test fails — stop and fix before continuing
```

## Процесс создания промпта

Когда пользователь просит создать goal-промпт:

1. **Уточни задачу** — что именно промпт должен делать, какие аргументы принимает, какой
   результат ожидается. Если пользователь уже описал задачу — используй это описание.

2. **Определи scope** — один файл, модуль, весь проект? От этого зависит бюджет и
   сложность workflow.

3. **Напиши frontmatter** — description, argument-hint. Description должен быть коротким
   но точным — он отображается в UI при выборе промпта.

4. **Напиши workflow** — пошаговая последовательность с вызовами goal-инструментов.
   Каждый шаг должен быть конкретным и проверяемым.

5. **Определи rules** — ограничения и правила, которые агент должен соблюдать при
   выполнении промпта. Это защитные перила — не делай X, остановись при Y.

6. **Укажи вывод** — куда сохранять результат, что делать после сохранения.

7. **Сохрани файл** в `prompts/goal_<name>.md` и покажи пользователю.

## Чеклист качества

Перед тем как отдать промпт пользователю, проверь:

- [ ] Frontmatter заполнен: description (обязательный), argument-hint (если нужны аргументы)
- [ ] Цель в create_goal — конкретная, с критериями успеха
- [ ] Workflow пошаговый, каждый шаг проверяем
- [ ] Есть completion audit перед update_goal
- [ ] Бюджет адекватен масштабу задачи
- [ ] Указано куда сохранять результат
- [ ] Нет противоречий между правилами
