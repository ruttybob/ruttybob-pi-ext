# Tool Ignore: Защита инструментов от деактивации через /tools

Status: wontfix (implemented)
Category: enhancement

## Problem Statement

Расширение `~/work/yapi/extensions/tools/` (команда `/tools`) позволяет переключать **любой** инструмент через `pi.setActiveTools()`. Это удобно, но некоторые инструменты не должны быть доступны для toggle — например, mesh-инструменты (`mesh_peers`, `mesh_send`, `mesh_reserve`, `mesh_release`, `mesh_manage`) критичны для координации агентов и их случайное отключение ломает mesh-протокол. Сейчас нет механизма исключения инструментов из UI toggle.

## Solution

Добавить конфигурационный файл `toolignore.json` с двумя уровнями: глобальный (`~/.pi/agent/toolignore.json`) и проектный (`./pi/toolignore.json`). Расширение `/tools` читает оба файла, объединяет списки исключённых инструментов, и не показывает их в SettingsList UI — они всегда остаются активными.

## User Stories

1. Как пользователь pi, я хочу чтобы mesh-инструменты всегда были активны и не появлялись в `/tools`, чтобы случайно не сломать координацию агентов
2. Как пользователь pi, я хочу прописать глобальные исключения в `~/.pi/agent/toolignore.json`, чтобы они применялись во всех проектах
3. Как пользователь pi, я хочу прописать проектные исключения в `./pi/toolignore.json`, чтобы они применялись только в конкретном проекте
4. Как пользователь pi, я хочу чтобы проектные исключения добавлялись к глобальным (union), а не перезаписывали их
5. Как пользователь pi, я хочу видеть в UI `/tools` только те инструменты, которые можно переключать, чтобы не путаться
6. Как пользователь pi, я хочу чтобы исключённые инструменты оставались активными всегда, независимо от действий в `/tools`
7. Как пользователь pi, я хочу чтобы при restore из session branch исключённые инструменты автоматически добавлялись в active set, даже если они не были сохранены в состоянии
8. Как владелец проекта, я хочу иметь возможность выключить любой инструмент через конфиг, добавив его имя в ignore-список
9. Как пользователь pi, я хочу чтобы отсутствие конфиг-файлов не ломало работу `/tools` — все инструменты доступны для toggle по умолчанию
10. Как пользователь pi, я хочу чтобы конфиг-файл поддерживал glob-паттерны (например `mesh_*`), чтобы не перечислять каждый инструмент вручную
11. Как пользователь pi, я хочу чтобы при некорректном конфиг-файле расширение показывало предупреждение и продолжало работать со всеми инструментами доступными

## Implementation Decisions

- **Формат конфига** — JSON файл `toolignore.json` с массивом строк. Каждая строка — точное имя инструмента или glob-паттерн (`*` совпадает с любой подстрокой). Пример:
  ```json
  ["mesh_*", "holo", "jot"]
  ```
- **Два уровня конфига**: глобальный `~/.pi/agent/toolignore.json` и проектный `<cwd>/.pi/toolignore.json`. Проектный **добавляется** к глобальному (union). При конфликте — оба применяются.
- **Размещение кода** — вся логика чтения конфига и фильтрации добавляется в `~/work/yapi/extensions/tools/index.ts`. Отдельных модулей не требуется — объём изменений минимальный.
- **Glob-матчинг** — простейший `fnmatch`-стиль: `*` заменяется на `.*`, `?` на `.`. Реализуется одной функцией `matchesPattern(name: string, pattern: string): boolean` без внешних зависимостей.
- **Фильтрация в SettingsList** — при построении `items` для SettingsList, каждый инструмент проверяется против объединённого ignore-списка. Если совпадает — инструмент пропускается (не добавляется в items).
- **Защита при restore** — в `restoreFromBranch()` после фильтрации сохранённых инструментов по `allToolNames`, добавить все инструменты из ignore-списка в `enabledTools`. Это гарантирует что игнорируемые инструменты всегда активны.
- **Обработка ошибок** — если файл не парсится как JSON, логируется предупреждение через `console.warn` и файл игнорируется (все инструменты доступны). Если файл не существует — это нормальное состояние, без предупреждений.
- **Начальное состояние** — `restoreFromBranch()` должен всегда убеждаться что игнорируемые инструменты присутствуют в `enabledTools` и в вызове `pi.setActiveTools()`.
- **Глобальный ignore по умолчанию** — в `~/.pi/agent/toolignore.json` пользователь сам решает что исключать. Расширение не создаёт файл автоматически и не поставляет дефолтный список.

## Testing Decisions

- **Glob-матчинг**: тесты для `matchesPattern()` — точное совпадение, `*` в начале/конце/середине, `?`, несколько паттернов. Чистая функция, легко тестировать.
- **Чтение конфига**: тесты для функции загрузки и объединения двух конфигов (глобальный + проектный). Проверить: оба существуют → union, только глобальный → глобальный, только проектный → проектный, ни одного → пустой массив, некорректный JSON → пустой массив + warning.
- **Фильтрация SettingsList items**: проверить что инструменты из ignore-списка не попадают в items, а остальные попадают.
- **Restore с ignore**: проверить что `restoreFromBranch()` всегда включает игнорируемые инструменты в `enabledTools`, даже если они не были сохранены в branch entry.
- Все тесты — на внешнее поведение (какой набор инструментов активен, какие items в UI), не на внутреннюю реализацию.
- Приоритет — модульные тесты утилитарных функций (matchesPattern, loadIgnoreConfig) и интеграционные тесты restore/filter логики.

## Out of Scope

- UI для редактирования ignore-списка (только ручное редактирование JSON)
- Проверка что инструмент с указанным именем реально существует (если указан несуществующий — просто игнорируется)
- Каскадное обновление: если пользователь вручную добавил mesh-инструмент в `/tools` до добавления в ignore — restore поправит при следующем событии
- Отдельная npm-зависимость для glob-матчинга — реализуем вручную
- Команда `/tools-ignore` или подобная для управления списком через UI

## Further Notes

- Mesh-инструменты по умолчанию должны быть в глобальном ignore у пользователя (`~/.pi/agent/toolignore.json` с `["mesh_*"]`). Это решение пользователя, а не расширения.
- Изменения вносятся только в `~/work/yapi/extensions/tools/index.ts` — ruttybob не затрагивается.
- Аналогичный паттерн (ignore/exclude) может быть полезен и для других расширений, которые управляют инструментами — но сейчас это out of scope.
- Существующий PRD `.scratch/zai-tools-toggle/PRD.md` описывает toggle zai-tools через отдельную команду `/zai-tools` — это ортогональная фича. Toolignore не заменяет toggle, а дополняет его, защищая критичные инструменты.
